## 用户故事

**作为** OneClawShadow 的用户，
**我想要** 在 AI 仍在回复或执行工具时发送额外的消息，
**以便** 我可以补充上下文、追问或重定向 AI，而无需等待当前回复完成。

### 典型场景

1. AI 正在流式输出一段很长的回复。用户意识到忘了提一个重要细节。用户直接输入并发送追加消息，无需等待。AI 完成当前回复后，立即看到追加消息并继续。

2. AI 正在执行一连串工具调用（例如读取多个文件）。在工具执行的间隙，用户发送"也检查一下配置文件"。AI 在下一轮迭代中收到这条消息并包含配置文件。

3. AI 正在流式回复。用户发送了一条新消息，然后改变主意按了停止。AI 的部分回复被保留。新消息留在聊天中但被标记为已废弃。用户随后发送第三条消息，AI 回应第三条消息，忽略废弃的那条。

4. 用户快速连续输入并发送两条消息，AI 还没开始回复。两条消息都出现在聊天中，AI 在第一次 API 调用时看到两条。

## 功能描述

### 功能概述

本功能移除了 AI agent loop 运行期间阻止用户发送消息的限制。当前实现中，streaming 期间发送按钮和输入框都被禁用。本功能实现后，用户可以随时输入和发送消息。在活跃的 agent loop 期间发送的消息会被排队，并在下一个自然暂停点注入（AI 回复完成和下一次 API 调用之间，或工具执行轮次之间）。AI 按正确的时间顺序将这些消息作为对话历史的一部分看到。

### 详细行为

#### Streaming 期间发送

- 输入框始终可用，无论 AI 是否在 streaming。
- 发送按钮始终可见，当输入框有文字时启用。
- 用户在 streaming 期间发送消息时：
  1. 消息立即作为用户气泡出现在聊天中（与普通用户消息样式相同）。
  2. 消息立即保存到数据库。
  3. 消息被保持在队列中，直到当前 AI 迭代完成。
  4. 在下一个迭代边界，AI 的上下文从数据库刷新，此时已包含排队的消息。
  5. AI 回应包含新消息在内的完整对话。

#### 停止按钮和发送按钮独立

- 停止按钮和发送按钮是两个独立的 UI 元素。
- Streaming / agent 执行期间：两个按钮同时可见。
  - **停止按钮**：出现在发送按钮左侧，带旋转进度指示器。点击取消当前 agent loop。
  - **发送按钮**：始终在右侧。输入框有文字且已配置 provider 时启用。
- 非 streaming 时：只有发送按钮可见（停止按钮消失）。

#### 有排队消息时按停止

用户在有排队（未处理）消息时按停止：

1. 当前 AI 回复被取消。部分回复保存到对话中。
2. 排队消息保留在聊天中（发送时已显示）。
3. 在排队消息之后插入一条系统提示到对话历史中。该提示指示 AI 忽略排队消息，回应用户的下一条消息。
4. Agent loop 终止。
5. 用户之后发送新消息时，AI 看到完整历史，包括部分回复、废弃的排队消息和系统提示。它回应新消息，而非废弃的消息。

#### 快速连发

如果用户在 agent loop 启动前发送了两条消息（例如在新对话中快速输入发送）：

- 两条消息都作为独立的 USER 消息保存到数据库。
- Agent loop 的第一次 API 调用从数据库读取两条消息。
- AI 看到两条连续的用户消息并回应两者。

### 用户交互流程

#### 流程 A：streaming 期间发送（正常）

```
1. 用户发送消息 A
2. AI 开始流式回复 A
3. 用户输入消息 B 并点击发送
4. 消息 B 立即出现在聊天中
5. AI 完成对 A 的回复
6. AI 自动开始回复 B（无需用户操作）
7. AI 对 B 的回复流式传入
```

#### 流程 B：工具执行期间发送

```
1. 用户发送消息 A
2. AI 开始回复，发起工具调用
3. 工具正在执行（显示旋转器）
4. 用户输入消息 B 并点击发送
5. 消息 B 立即出现在聊天中
6. 工具执行完成
7. AI 在下一轮迭代中看到工具结果 + 消息 B
8. AI 综合考虑工具结果和消息 B 进行回复
```

#### 流程 C：带排队消息的停止

```
1. 用户发送消息 A
2. AI 开始流式回复 A
3. 用户输入消息 B 并点击发送
4. 消息 B 立即出现在聊天中
5. 用户点击停止按钮
6. AI streaming 停止。部分回复在聊天中可见。停止按钮消失。
7. 消息 B 在部分回复下方可见
8. 系统提示被插入（对用户不突出显示）
9. 用户之后发送消息 C
10. AI 回应 C（忽略 B）
```

#### 流程 D：快速连发

```
1. 用户输入消息 A 并点击发送
2. 用户立即输入消息 B 并点击发送
3. A 和 B 都出现在聊天中
4. AI 启动并在上下文中看到 A 和 B
5. AI 综合回应 A 和 B 的上下文
```

## 验收标准

必须满足（全部要求）：
- [ ] 输入框在 streaming 期间始终可用（用户可随时输入）
- [ ] 发送按钮始终可见，文字非空时启用，即使在 streaming 期间
- [ ] 停止按钮和发送按钮在 streaming 期间是独立的 UI 元素
- [ ] 停止按钮在 streaming 期间显示旋转进度指示器
- [ ] Streaming 期间发送的消息立即作为用户气泡出现在聊天中
- [ ] Streaming 期间发送的消息在当前迭代完成后被 AI 处理
- [ ] AI 对排队消息的回复自动开始（无需用户操作）
- [ ] 有排队消息时按停止：部分 AI 回复被保存，排队消息保持可见，系统提示被插入
- [ ] 带排队消息停止后，AI 回应用户的下一条消息（排队消息被忽略）
- [ ] AI 启动前快速连发：两条消息都出现，AI 回应两者
- [ ] 无 UI 状态损坏：上述所有流程后，聊天保持一致可用状态

可选（锦上添花）：
- [ ] 显示排队消息数量的角标或指示器
- [ ] 排队消息在处理前有细微视觉区分（例如轻微透明度）

## UI/UX 要求

### Streaming 期间的输入区域布局

```
+--------------------------------------------------+
| [输入框 - 始终可用                           ]    |
|                          [停止旋转] [发送]        |
+--------------------------------------------------+
```

- 输入框：全宽，始终接受输入。
- 停止按钮：`CircularProgressIndicator` 图标，`errorContainer` 背景。仅在 streaming 时可见。
- 发送按钮：`Send` 图标，`primary` 背景。始终可见。仅在文字为空或未配置 provider 时禁用。

### 非 Streaming 时的输入区域布局

```
+--------------------------------------------------+
| [输入框                                      ]    |
|                                        [发送]    |
+--------------------------------------------------+
```

- 与当前行为相同，去掉 `canSend` / `isStreaming` 禁用。

### 消息显示

- 排队消息与普通用户消息外观完全相同。V1 不需要特殊样式。
- 系统提示（废弃标记）使用现有的 `SystemMessageCard` 样式（小灰色居中文字）。

### 交互反馈

- Streaming 期间发送：消息即时出现，与正常发送相同。不需要额外的 toast 或动画。
- 带排队消息的停止：停止行为与当前相同（部分回复保存）。系统提示是聊天中的小灰色文字标签，不是弹窗或 toast。

## 功能边界

### 包含的功能
- Streaming 期间始终可用的输入框
- 独立的停止和发送按钮
- 基于队列的消息注入（在迭代边界）
- 停止 + 排队消息的废弃系统提示处理
- 快速连发支持

### 不包含的功能
- 在 streaming 的 token 中途打断以注入消息（仅在迭代边界注入）
- 编辑或取消已发送的排队消息
- 排队与已处理消息的视觉区分（可选，非必需）
- 多个并发 agent loop

## 业务规则

### 消息规则
1. 用户可以在回复生成期间发送新消息（变更 FEAT-001 规则 3）。
2. Streaming 期间发送的消息被排队并在下一个迭代边界处理。
3. AI 按时间顺序将排队消息作为对话历史的一部分看到。
4. 排队消息永不丢失——用户发送的瞬间就保存到数据库。

### 停止规则
1. 有排队消息时按停止，部分 AI 回复被保存（不丢弃）。
2. 停止后排队消息保留在数据库和聊天 UI 中。
3. 在排队消息之后插入系统提示，指示 AI 忽略它们。
4. 系统提示在聊天中作为小型系统消息可见。

### 队列规则
1. 可排队的消息数量没有限制。
2. 消息按发送顺序注入。
3. 如果 agent loop 自然结束（非停止）且有排队消息，所有排队消息被注入，AI 回应它们。

## 测试要点

### 功能测试 -- 用户操作 APP

这些测试描述用户在设备上的操作和预期看到的结果。它们是本功能的主要验证方式。

#### 测试 1：AI streaming 期间发送消息

**前置条件**：已配置 provider，聊天界面打开。

| 步骤 | 用户操作 | 预期结果 |
|------|----------|----------|
| 1 | 输入"给我讲一个关于龙的长故事"并点击发送 | 消息 A 出现为用户气泡。AI 开始流式回复。停止按钮（旋转）出现在发送按钮旁边。 |
| 2 | AI 正在 streaming 时，在输入框输入"要搞笑的" | 输入框接受输入。发送按钮启用。 |
| 3 | 点击发送 | "要搞笑的"作为第二个用户气泡出现在 streaming AI 回复下方。输入框清空。 |
| 4 | 等待 AI 完成当前回复 | AI 对消息 A 的回复完成。 |
| 5 | 观察 | AI 自动开始新的流式回复，回应"要搞笑的"。无需用户操作。 |

#### 测试 2：工具执行期间发送消息

| 步骤 | 用户操作 | 预期结果 |

| 1 | 输入"现在几点了？"并点击发送 | 消息出现。AI 发起工具调用。工具调用卡片显示旋转器。 |
| 2 | 工具执行期间，输入"今天几号？"并点击发送 | 新用户气泡立即出现。 |
| 3 | 等待工具完成和 AI 回复完成 | AI 完成对第一个查询的回复。然后 AI 处理第二条消息（可能再次调用工具获取日期）。 |

#### 测试 3：带排队消息的停止

| 步骤 | 用户操作 | 预期结果 |

### 边界情况

- AI streaming 期间连续发送 5 条消息（验证 5 条都出现，AI 最终回应所有消息）
- 没有排队消息时按停止（验证：与当前行为相同，不插入系统提示）
- 发送消息后 AI 出错，再发送一条消息（验证：错误消息可见，新消息被处理）
- 非常长的排队消息（10,000+ 字符）
- Streaming 期间空输入（验证：发送按钮保持禁用，点击无反应）
