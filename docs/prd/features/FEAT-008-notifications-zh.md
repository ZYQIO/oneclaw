## 用户故事

**作为**一个已启动长时间运行 AI 任务的用户，
**我希望**在任务完成或失败时收到通知，
**以便**我可以在等待期间切换到其他应用，并在 AI 完成时返回。

### 典型场景

1. 用户向 AI 提出一个涉及多次工具调用的复杂问题。用户切换到其他应用。几分钟后，一条通知出现："Task completed -- Here's a summary of the files I found..."。用户点击通知，返回到会话。

2. 用户要求 AI 执行某项操作，然后切换应用。API 请求因速率限制而失败。一条通知出现："Task failed -- Rate limit exceeded. Please try again later."。用户点击通知查看错误信息。

3. 用户正在应用内观看 AI 流式输出。不显示任何通知 -- 界面已经可见。

## 功能描述

### 概述

当 AI agent 任务在应用处于后台时完成或失败，应用会发送 Android 推送通知。通知包含结果预览，点击通知可导航回相关会话。当用户正在查看应用时，不发送通知。

### 通知发送时机

当以下条件**全部**满足时发送通知：
1. 应用处于**后台**（用户已切换到其他应用或屏幕已关闭）。
2. Agent 循环已**完成** -- 成功完成 (ResponseComplete) 或发生错误 (ChatEvent.Error)。

以下情况**不**发送通知：
- 应用处于前台（用户正在查看聊天界面）。
- Agent 循环仍在运行（等待 API 响应、执行工具、流式传输中）。

### 通知内容

#### 任务完成

```
OneClaw Shadow
Task completed
[AI 回复预览，前 ~100 个字符]
```

- 标题: "Task completed"
- 正文: AI 最终回复文本的前 ~100 个字符，超出部分以 "..." 截断
- 点击操作: 打开应用并导航到相关会话

#### 任务失败

```
OneClaw Shadow
Task failed
[错误信息，前 ~100 个字符]
```

- 标题: "Task failed"
- 正文: ChatEvent.Error 中的错误信息，超出部分以 "..." 截断
- 点击操作: 打开应用并导航到相关会话

### Notification Channel

单个 Android notification channel：
- Channel ID: `agent_tasks`
- Channel 名称: "Agent Tasks"
- 重要性: Default（声音 + 通知栏显示）
- 自动遵守系统勿扰模式设置

### 前台检测

应用使用 `ProcessLifecycleOwner`（来自 `androidx.lifecycle`）跟踪是否处于前台。当接收到 `ON_STOP` 事件时，应用处于后台；当接收到 `ON_START` 事件时，应用回到前台。

## 验收标准

必须通过（全部为必需项）：
- [ ] 当应用处于后台时，AI 任务完成后显示通知
- [ ] 当应用处于后台时，AI 任务失败后显示通知
- [ ] 当应用处于前台时，不显示通知
- [ ] 通知包含 AI 回复或错误信息的预览
- [ ] 点击通知打开应用并导航到正确的会话
- [ ] 通知使用正确的 Android notification channel
- [ ] 通知遵守系统勿扰模式设置
- [ ] 任务仍在进行中（等待/流式传输）时不显示通知

可选项（锦上添花）：
- [ ] 多个任务完成时的通知分组
- [ ] 通知操作（例如直接从通知中"回复"）

## UI/UX 要求

### 通知外观

标准 Android 通知外观：
- 小图标: 应用图标（单色）
- 大图标: 无（使用默认）
- 标题: "Task completed" 或 "Task failed"
- 正文: 预览文本（~100 个字符）
- 点击: 打开应用到相关会话

### 无应用内通知 UI

当应用处于前台时，不显示 toast、snackbar 或浮层通知。聊天界面本身即为实时反馈。

## 功能边界

### 包含
- 任务完成时的推送通知（后台）
- 任务失败时的推送通知（后台）
- 包含结果/错误预览的通知
- 点击导航到相关会话
- 单个 notification channel

### 不包含
- 应用内通知浮层或 toast
- 设置中的通知偏好/开关（V1 版本：应用处于后台时始终开启）
- 多个 notification channel（例如错误单独一个 channel）
- 自定义通知声音或振动模式
- 同步状态通知 (FEAT-007)
- 非聊天事件的通知

## 业务规则

1. 通知仅在应用处于后台时发送。
2. 每个完成/失败的任务发送一条通知。如果多个任务完成，每个任务分别发送通知。
3. 通知预览文本截断为 ~100 个字符。
4. 点击通知始终导航到任务所在的会话。
5. 如果用户清除通知，不采取进一步操作（不会重新通知）。

## 测试要点

### 功能测试 -- 用户操作应用

#### 测试 1: 任务完成时的通知

| 步骤 | 用户操作 | 预期结果 |
|------|----------|----------|
| 1 | 向 AI 发送一条消息 | AI 开始流式输出。 |
| 2 | 按 Home 键（切换到其他应用） | 应用进入后台。AI 继续处理。 |
| 3 | 等待 AI 完成 | 通知栏中出现一条通知："Task completed"，并附有回复预览。 |
| 4 | 点击通知 | 应用打开并导航到会话，已完成的回复可见。 |

#### 测试 2: 任务失败时的通知

| 步骤 | 用户操作 | 预期结果 |
|------|----------|----------|
| 1 | 配置一个使用无效 API key 的 provider | Provider 设置了错误的 key。 |
| 2 | 发送一条消息 | AI 开始处理（将会失败）。 |
| 3 | 按 Home 键 | 应用进入后台。 |
| 4 | 等待 API 调用失败 | 出现一条通知："Task failed"，并附有错误信息。 |
| 5 | 点击通知 | 应用打开，导航到会话，聊天中可见错误信息。 |

#### 测试 3: 应用在前台时不显示通知

| 步骤 | 用户操作 | 预期结果 |
|------|----------|----------|
| 1 | 留在应用内发送一条消息 | AI 流式输出回复。 |
| 2 | 等待完成 | 回复出现在聊天中。通知栏无通知。 |

#### 测试 4: 通知导航到正确的会话

| 步骤 | 用户操作 | 预期结果 |
|------|----------|----------|
| 1 | 打开 Session A，发送一条消息 | AI 在 Session A 中开始处理。 |
| 2 | 通过抽屉菜单切换到 Session B | Session B 现在可见。 |
| 3 | 按 Home 键 | 应用进入后台。Session A 的任务继续运行。 |
| 4 | 等待 Session A 的任务完成 | 出现 Session A 的通知。 |
| 5 | 点击通知 | 应用打开到 Session A（而非 Session B）。 |

### 边界情况

- 多个任务在后台完成：每个任务分别获得一条通知
- 用户清除所有通知：对应用无副作用
- 任务运行中应用被杀死：任务丢失（无通知，与当前行为一致）
- 通知权限被拒绝（Android 13+）：通知静默失败，应用仍正常工作
