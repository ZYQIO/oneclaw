## 用户故事

**作为** OneClawShadow 的用户，
**我想要** 发送文字消息给 AI 模型并实时看到它的响应（包括它所做的任何工具调用），
**以便** 我可以在手机上与一个能代我执行操作的 AI Agent 进行交互。

### 典型场景
1. 用户打开一个会话，输入一个问题，收到流式 AI 响应。
2. 用户要求 AI 执行一个需要工具调用的任务（例如"现在几点了？"）。AI 调用工具，结果在聊天中展示，AI 给出最终回答。
3. 用户在长对话中向上滚动查看较早的消息。
4. 用户在生成过程中停止响应，因为 AI 的方向不对。

## 功能描述

### 概述
对话交互是 OneClawShadow 的核心界面。它提供一个基于聊天的对话视图，用户发送文字消息给 AI 模型并接收流式响应。当 AI 模型发起工具调用时，执行过程和结果会内联显示在对话中。该模块处理完整的请求-响应循环，包括流式传输、工具调用编排展示和错误状态。

### 消息类型

聊天中展示以下类型的消息：

1. **用户消息**：用户输入的文字。显示在聊天的右侧。

2. **AI 响应**：模型的文字回复。显示在聊天的左侧。支持 Markdown 渲染。可能包含附属的思考/推理块（见下文）。

3. **思考块**：模型的内部推理过程（来自扩展思考/思维链）。作为 AI 响应消息的可折叠部分展示，而非独立消息。默认折叠；用户可以点击展开查看推理过程。

4. **工具调用消息**：当 AI 调用工具时展示。显示工具执行生命周期（调用、执行、结果）。有两种展示模式：
   - **简洁模式**：单行文字，如"正在调用 [工具名]..."，然后"完成"加结果摘要。
   - **详细模式**：展示工具名、输入参数和完整返回值。可展开/折叠。
   - 用户可以全局切换这两种模式。

5. **错误消息**：当出现问题时展示——API 请求失败、工具执行错误、网络超时等。显示清晰的错误描述。

### 核心交互

#### 发送消息
- 屏幕底部的文字输入框
- 发送按钮（仅在输入非空且没有正在进行的请求时启用）
- 发送后，用户消息立即出现在聊天中
- 发送后输入框清空

#### 接收响应
- AI 响应逐 token（或逐块）流式传入
- 等待第一个 token 时显示打字/加载指示器
- 文字随着到达逐步渲染
- 响应流式传入时实时渲染 Markdown

#### 工具调用流程
当模型的响应包含工具调用时：
1. 工具调用消息出现在聊天中（根据当前模式显示简洁或详细）
2. 工具自动执行（默认不需要用户确认）
3. 工具结果展示
4. 结果发送回模型
5. 模型继续响应（可能包含更多工具调用或最终文字回答）
6. 此循环持续直到模型产生没有更多工具调用的最终响应

#### 停止生成
- 响应正在生成时，停止按钮替代发送按钮
- 点击停止取消进行中的请求
- 已接收的内容保留在聊天中
- 点击停止后，停止按钮**必须**立即切换回发送按钮——UI 绝不能在用户点击停止后仍然停留在"流式传输中"状态
- 用户随后可以发送新消息

#### 复制消息
- 所有消息（用户消息、AI 响应、工具调用详情、错误消息）都可以复制
- 长按消息复制其文字内容
- 对于包含 Markdown 的 AI 响应，复制提供原始文本

#### 滚动和历史
- 聊天垂直滚动，最新消息在底部
- 新消息到达或流式传输期间自动滚动到底部
- 用户可以向上滚动查看更早的消息
- 在流式传输期间向上滚动时，出现"滚动到底部"按钮

### 工具调用展示模式

#### 简洁模式
```
[图标] 正在调用 read_file... 完成
```
每个工具调用一行。显示工具名和状态。结果隐藏，除非用户点击展开。

#### 详细模式
```
[图标] 工具: read_file
  输入: { "path": "/storage/notes.txt" }
  结果: "文件内容..."
[可折叠]
```
显示完整的工具名、输入参数和返回值。每个部分可折叠。

#### 模式切换
- 聊天界面中的切换按钮（例如工具栏图标）允许在简洁和详细模式之间切换
- 设置适用于当前视图中的所有工具调用消息
- 偏好设置会被持久化

## 验收标准

必须通过（所有必需项）：
- [ ] 用户可以输入并发送文字消息
- [ ] AI 响应以流式方式展示（逐步 token 渲染）
- [ ] AI 响应中的 Markdown 正确渲染（粗体、斜体、列表、标题、行内代码、代码块）
- [ ] 思考/推理块默认折叠展示，点击可展开
- [ ] 当前模型支持 extended thinking 时，自动启用（无需用户操作）——响应中出现"正在思考..."块，无需任何手动配置
- [ ] 工具调用内联显示在对话中
- [ ] 工具调用简洁模式在一行内显示工具名和状态
- [ ] 工具调用详细模式显示工具名、参数和结果，可折叠
- [ ] 用户可以在简洁和详细工具调用展示模式之间切换
- [ ] 展示模式偏好跨会话持久化
- [ ] 等待第一个响应 token 时显示打字/加载指示器
- [ ] 用户可以停止正在进行的响应生成
- [ ] 停止后保留已接收的部分内容
- [ ] 点击停止后停止按钮立即切换回发送按钮（UI 不得停留在流式传输状态）
- [ ] 所有消息类型都可以复制（长按复制）
- [ ] API 调用或工具执行失败时清晰地显示错误消息
- [ ] 新消息和流式传输时聊天自动滚动到底部
- [ ] 向上滚动时出现"滚动到底部"按钮
- [ ] 输入为空或请求进行中时发送按钮禁用
- [ ] 软键盘弹出时，顶部应用栏（汉堡图标和设置齿轮）保持完整可见，不被推出屏幕

可选（V1 中的加分项）：
- [ ] 发送时触觉反馈
- [ ] 平滑滚动动画
- [ ] 消息时间戳显示

## UI/UX 要求

详细的视觉规格、布局、间距和组件设计请参见 [UI 设计规范](../../design/ui-design-spec-zh.md)。以下是关键点摘要：

### 布局
- 全屏聊天视图，导航抽屉中放置会话列表（通过汉堡菜单访问）
- 顶部应用栏：汉堡菜单（左）、带下拉箭头的 Agent 选择器（中）、设置齿轮（右）
- 消息列表占据屏幕大部分空间
- 输入区域固定在底部：pill 形状文字输入框 + 发送/停止按钮
- 软键盘弹出时，**只有消息列表和输入区域进行调整**——顶部应用栏必须始终完整可见。汉堡图标和设置齿轮绝不能被键盘推出屏幕外。实现方式是使用 `WindowCompat.setDecorFitsSystemWindows(false)` 配合 Compose 的 `imePadding()`，而不是依赖 `windowSoftInputMode=adjustResize`。

### 消息气泡
- 用户消息：右对齐，`primaryContainer` 背景，圆角（18dp）
- AI 消息：左对齐，无气泡/无背景，左侧小 AI 图标
- 每条完成的 AI 响应下方有操作行：复制、重新生成（可选）、分享（可选）
- 工具调用消息：全宽，`surfaceVariant` 背景，视觉上有区分
- 错误消息：全宽，`errorContainer` 背景

### 思考块
- 附在 AI 消息气泡上，在主要响应文本上方
- 折叠状态：小标签如"正在思考..."或"查看推理过程"，带展开图标
- 展开状态：以视觉上有区分的样式显示完整思考文本（例如更浅的文字颜色、不同背景）
- **对支持 extended thinking 的模型（如 Claude Opus 4、Claude Sonnet 4 系列），默认自动启用扩展思考模式。** 应用会自动检测模型能力，并在 API 请求中加入所需的思考配置，无需用户手动开启——支持 thinking 的模型始终处于 thinking 开启状态。

### 流式指示器
- 等待第一个 token 时：在 AI 消息区域显示脉冲点或打字指示器
- 流式传输中：AI 消息文本逐步增长

### 交互反馈
- 加载：等待响应时的打字指示器
- 错误：聊天中显示错误消息，带重试选项
- 成功：消息自然出现在聊天流中
- 停止：生成停止，部分内容保留

## 业务规则

### 消息规则
1. 用户消息始终为纯文本（用户侧不做 Markdown 渲染）
2. AI 响应支持 Markdown 渲染
3. 响应正在生成时用户不能发送新消息（必须先停止）
4. 不能发送空消息
5. 工具调用默认自动执行，无需用户确认（需要 Android 系统权限时除外）

### 展示规则
1. 消息按时间顺序展示，最早的在顶部
2. 工具调用展示模式（简洁/详细）是全局设置，非按消息设置
3. 思考块在首次展示时始终默认折叠
4. 错误消息在对话中错误发生的位置展示
5. 对支持 thinking 的模型，扩展思考在每次 API 请求中始终启用——V1 中不提供关闭 thinking 的用户开关。

### 数据规则
1. 会话中的所有消息持久化存储在本地
2. 消息包含元数据：时间戳、消息类型、token 数量（如可用）
3. 工具调用消息包含：工具名、输入参数、输出结果、执行状态、执行时长

## 非功能性需求

### 性能
- 流式响应每个块的渲染延迟 < 50ms
- 滚动浏览 1000+ 条消息应保持 60fps
- 输入框对按键响应无可感知延迟
- 工具调用执行不应阻塞 UI（在后台线程运行）

### 可靠性
- 如果流式响应中断（网络断开），展示已接收的内容并显示错误
- 如果工具调用失败，在工具调用消息中显示错误并让模型处理
- 应用不应因格式错误的 API 响应而崩溃

### 无障碍
- 消息应支持系统字体大小设置
- 所有消息类型有足够的颜色对比度
- 支持屏幕阅读器读取消息内容

## 错误处理

### 错误场景

1. **网络不可用**
   - 展示：聊天中的错误消息"无法连接。请检查您的网络并重试。"
   - 用户操作：可以再次发送消息重试

2. **API 请求失败（4xx/5xx）**
   - 展示：聊天中的错误消息，包含错误详情（例如"API 错误：429 请求频率超限"）
   - 用户操作：可以再次发送消息重试

3. **API key 无效或已过期**
   - 展示：聊天中的错误消息"API key 无效或已过期。请检查您的 Provider 设置。"
   - 用户操作：导航到 provider 设置进行修复

4. **工具执行失败**
   - 展示：错误在工具调用消息块中显示
   - 错误作为工具结果发送回模型，让模型决定如何继续

5. **流式传输中断**
   - 展示：保留部分响应，附加错误消息
   - 用户操作：可以发送新消息继续

6. **响应超时**
   - 展示：聊天中的错误消息"请求超时。请重试。"
   - 用户操作：可以重试

## 测试要点

### 功能测试
- 发送消息并验证其出现在聊天中
- 验证流式响应逐步渲染
- 验证 Markdown 正确渲染（测试各种元素）
- 验证思考块默认折叠并可点击展开
- 验证工具调用简洁模式展示
- 验证工具调用详细模式展示
- 验证模式切换功能并持久化
- 验证停止生成功能并保留部分内容
- 验证所有消息类型的复制功能
- 验证自动滚动行为
- 验证向上滚动时出现"滚动到底部"按钮
- 验证每种错误场景的错误消息正确展示
- 验证发送按钮状态（空输入时禁用、生成中禁用）

### 性能测试
- 流式延迟测量
- 大量消息历史（1000+ 条）的滚动性能
- 长对话期间的内存使用

### 边界情况
- 非常长的消息（10,000+ 字符）
- 快速连续发送消息
- 单次响应中的多个工具调用
- 嵌套工具调用（工具调用结果触发另一个工具调用）
- 空 AI 响应
- 流式传输期间网络切换（WiFi 到蜂窝）
