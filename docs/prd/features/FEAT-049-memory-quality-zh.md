## 用户故事

**作为** OneClawShadow 的用户，
**我希望** AI 只将有意义、无冗余的信息保存到长期记忆中，
**以便** MEMORY.md 能够保持简洁、高信噪比的知识库，而不是不断积累噪音（例如重复的模型切换记录、重复条目和无关紧要的观察）。

### 痛点（当前状态）

`save_memory` 工具（FEAT-023）机械地工作——它会追加 AI 决定保存的任何内容。然而，工具描述对于"什么是重要信息"过于模糊，导致以下问题：

1. **瞬态状态污染** -- AI 保存每一次模型配置变更（例如"用户从 Opus 切换到 Haiku"），一周内产生 6 条以上几乎相同的条目
2. **重复积累** -- 同一偏好（例如"用户想清理促销邮件"）因为 AI 从不检查已有内容，以不同措辞被保存了 3 次
3. **低价值观察** -- 截图中随机的视觉细节（例如"用户有一块色彩鲜艳的花纹地毯"）被永久存储
4. **无更新机制** -- 所有写入都是追加式的；过时的事实（例如旧的模型偏好）永远无法被修正，只会不断叠加

### 典型场景

1. 用户将默认模型从 Sonnet 切换到 Haiku。AI **不应该**保存此操作——这是瞬态配置，而非长期偏好。
2. 用户说"记住我在所有应用中都偏好深色模式"。AI 将其保存。之后用户说"其实我现在更喜欢浅色模式了"。AI 应该**更新**已有条目，而不是追加一条矛盾的新内容。
3. 用户进行了多次关于清理 Gmail 收件箱的对话。AI 在多次对话后识别出规律："用户需要邮件自动化工具"。它只保存一条简洁摘要，而不是三条重复内容。
4. 用户分享了一张截图。除非被明确要求，AI **不应该**保存关于家具或房间装饰的观察。

## 功能描述

### 概述

本功能通过三个递进阶段改善记忆质量：

1. **第一阶段 -- 提示词防护栏**：用包含明确保存/跳过标准、反面示例和去重检查要求的新描述重写 `save_memory` 工具说明
2. **第二阶段 -- 先读后写**：添加一种机制使 AI 在保存前能查看现有 MEMORY.md 内容，并新增 `update_memory` 工具用于修改已有条目
3. **第三阶段 -- 结构化记忆与压缩**：将 MEMORY.md 组织为语义分区，并添加周期性压缩功能以合并重复项和删除过期条目

### 详细规格

#### 第一阶段：提示词防护栏（`save_memory` 工具描述重写）

用精确的新描述替换当前模糊的工具描述，其中包含：

**正向触发条件**（何时保存）：
- 用户明确说"记住这个"/"保存这个"
- 经 2 次以上对话确认的稳定用户偏好
- 重要的个人背景（职业、关键项目、领域专长）
- 多次观察到的重复性工作流模式

**负向触发条件**（何时不保存）：
- 瞬态状态：当前模型选择、临时设置、会话专属配置
- 一次性观察：截图内容、单次请求、视觉环境细节
- 频繁变化的信息（例如"目前正在做 X"）
- MEMORY.md 中已存在的信息（保存前必须检查）
- 从单次交互中推断的个性特征

**保存前检查清单**（嵌入工具描述中）：
1. 这条信息 30 天后还有用吗？
2. 现有记忆中是否已有此内容？如有，跳过或更新。
3. 这是已确认的规律，还是只是一次性事件？

#### 第二阶段：先读后写 + 更新机制

两项变更：

1. **在 save_memory 响应中注入现有记忆**：调用 `save_memory` 时，工具在返回成功消息的同时返回当前 MEMORY.md 内容，以便 AI 查看已有内容。或者，工具描述指示 AI 在调用 save 之前先使用 `read_memory`（或注入的系统提示词记忆）。

2. **新增 `update_memory` 工具**：受 Mem0 的 ADD/UPDATE/DELETE/NOOP 分类以及 Letta 的 `memory_replace`/`memory_rethink` 工具启发。该工具允许 AI：
   - 用更新内容替换特定已有条目
   - 删除过期条目
   - AI 提供旧文本（或唯一子字符串）以及新文本

#### 第三阶段：结构化记忆与压缩

1. **分区 MEMORY.md**：将记忆组织为语义分类：
   ```markdown
   # Long-term Memory

   ## User Profile
   - Software engineer, maker/builder personality

   ## Preferences
   - Prefers dark mode in all apps
   - Creative writing: prefers longer-form content

   ## Interests
   - Card game: Sushi Go
   - Stock price data retrieval

   ## Workflow
   - Gmail: wants promotional email cleanup automation
   - Screenshots: prefers 900x2048, adequate JS render wait time
   ```

2. **记忆压缩与备份**：一个后台进程（在应用进入后台时触发，或手动触发），它将：
   - 在任何破坏性操作之前创建 MEMORY.md 的带时间戳备份（例如 `MEMORY_backup_2026-03-02_14-30-00.md`）
   - 读取全部 MEMORY.md 内容
   - 使用 LLM 合并重复项、删除矛盾（保留最新版本），并重新组织为分区格式
   - 用压缩后的版本覆盖 MEMORY.md
   - 清理旧备份（保留最近 5 份）
   - 重建搜索索引
   - 如果压缩出现问题，用户（或开发者）可从备份中恢复

3. **带分类参数的 save_memory**：添加可选的 `category` 参数，使 AI 能将新记忆放入正确的分区。

## 功能边界

### 包含范围
- 重写的 `save_memory` 工具描述（第一阶段）
- 新增 `update_memory` 内置工具（第二阶段）
- 带分区标题的结构化 MEMORY.md（第三阶段）
- 记忆压缩逻辑（第三阶段）
- `save_memory` 上的可选 `category` 参数（第三阶段）

### 不包含范围
- 知识图谱或实体关系提取（Zep/Graphiti 风格）——对当前需求过于复杂
- 写入时基于嵌入的自动去重——延至未来优化
- 完整的记忆版本控制或撤销历史 UI（但为安全起见包含压缩前备份）
- 跨设备记忆同步
- 记忆导入/导出

## 业务规则

### 记忆保存规则（第一阶段）
1. 工具描述是主要的质量控制机制——AI 必须根据提示词进行自我约束
2. 不应用程序化过滤；信任 LLM 遵循描述
3. 5,000 字符限制和非空验证与 FEAT-023 保持不变

### 记忆更新规则（第二阶段）
1. `update_memory` 要求 `old_text` 参数在 MEMORY.md 中精确匹配
2. 如果未找到 `old_text`，工具返回错误（不支持部分匹配）
3. 如果 `new_text` 为空/null，则删除该条目（移除 old_text）
4. 匹配前对 old_text/new_text 周围的空白字符进行修剪
5. 每次工具调用只进行一处替换（不支持全局替换）

### 记忆压缩规则（第三阶段）
1. 每次压缩**之前**必须创建 MEMORY.md 的带时间戳备份（无例外）
2. 备份保留：保留最近 5 份备份；每次压缩后自动清理更旧的备份
3. 压缩使用与当前聊天会话相同的 LLM 模型
4. 压缩有 token 预算限制以防止过高的 API 费用
5. 压缩保留矛盾事实的最新版本
6. 压缩不会删除用户明确要求记住的条目（带有"User requested:"前缀的条目）
7. 自动压缩的大小阈值：MEMORY.md 超过 3,000 字符
8. 如果 LLM 返回空结果或可疑的过短结果，压缩将被中止并保留原始内容（备份仍然存在作为额外保障）

## 错误处理

### 错误场景

1. **`update_memory` 目标文本未找到**
   - 工具返回：`ToolResult.error("not_found", "The specified text was not found in MEMORY.md. Use read_memory to check current content.")`
   - AI 报告错误，并可用修正后的文本重试

2. **`update_memory` 目标文本匹配多处**
   - 工具返回：`ToolResult.error("ambiguous_match", "The specified text matches multiple locations. Provide more surrounding context to make the match unique.")`
   - AI 提供更多上下文后重试

3. **记忆压缩 LLM 调用失败**
   - 处理方式：原始 MEMORY.md 被保留（备份在 LLM 调用之前已创建）
   - 用户影响：无；下次触发时重试压缩

4. **记忆压缩产生空结果**
   - 处理方式：拒绝该结果并保留原始内容；备份仍然可用
   - 记录警告日志以便调试

5. **记忆压缩产生不良结果（数据丢失）**
   - 处理方式：压缩前备份允许通过 `MemoryFileStorage.restoreFromBackup()` 进行恢复
   - 未来：记忆设置界面将提供"从备份恢复"选项

5. **压缩期间 MEMORY.md 被锁定**
   - 处理方式：使用 Mutex 防止压缩期间的并发写入
   - 压缩期间的 `save_memory` 和 `update_memory` 调用等待压缩完成

## 行业背景

本设计参考了对现有 AI 记忆系统的研究：

- **Mem0**：两阶段提取-更新流程，采用 ADD/UPDATE/DELETE/NOOP 分类（第二阶段借鉴自此）
- **Letta (MemGPT)**：代理通过 `memory_replace`/`memory_rethink` 工具自主管理记忆（第二阶段的 `update_memory` 受此启发）
- **LangChain/LangMem**：区分语义/情节/程序记忆；使用合并更新的 Profile 对象（第三阶段的结构化分区借鉴了此概念）
- **Anthropic Claude Memory Tool**：基于文件的方法，使用 `str_replace` 进行更新；系统提示词指示代理在行动前检查记忆（第一和第二阶段与此对齐）
- **Zep/Graphiti**：具有边缘失效机制处理矛盾的时序知识图谱（对我们的需求过于复杂，但"失效而非删除"原则为第三阶段的压缩提供了参考）

所有系统的核心洞见：**记忆的质量由写入时决定，而非读取时**。在信息持久化之前进行过滤和结构化，远比在检索时试图清理嘈杂的记忆存储更为有效。
