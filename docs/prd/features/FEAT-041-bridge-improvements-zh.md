## 用户故事

**作为** OneClaw 消息桥接的用户，
**我希望** 桥接功能能够可靠运行，具备正确的格式、适当的反馈以及无缝的会话集成，
**以便** 我通过 Telegram 及其他平台与 AI 智能体聊天的体验自然流畅、精致完善。

### 典型场景
1. 用户通过 Telegram 发送消息。AI 回复以简洁的格式呈现——段落之间只有一个空行，列表项紧凑无多余间距，引用块使用 Telegram 原生样式。
2. 用户发送消息后，立即在 Telegram 中看到"正在输入..."指示器，在智能体处理过程中提供实时反馈。
3. 用户在应用中切换到不同的会话，然后通过 Telegram 发送消息。桥接消息出现在刚切换到的会话中，而非单独的桥接专用会话。
4. 用户通过 Telegram 发送 `/clear`。系统创建一个新会话，并将其设为后续桥接消息的活跃会话。
5. 手机重启。桥接服务自动启动并恢复运行，无需用户干预。
6. 用户在应用中打开桥接设置，看到清晰、有序的界面，包含可折叠的频道区块、设置指南和配置选项。

## 功能描述

### 概述
FEAT-041 根据用户测试反馈，整合了对消息桥接（FEAT-024）的多项改进。变更涵盖三个类别：格式质量、交互反馈和会话管理。此外，还纳入了近期开发中的若干可靠性和用户体验改进。

### 改进内容

#### 1. Telegram 消息格式（修复）
- **问题**：由于基于正则表达式的 HTML 转换逻辑，AI 在 Telegram 中的回复在段落、标题和列表项之间存在过多空行。
- **解决方案**：将 `TelegramHtmlRenderer` 从基于正则表达式的 HTML 转换重写为 AST 访问者模式，直接遍历 commonmark 解析树。智能的 `appendBlockSeparator` 方法仅在兄弟块之间添加换行，消除多余空白。
- **修复前**：每个段落、标题和列表项之间存在多个空行。
- **修复后**：顶级块之间只有一个空行，列表项紧凑，使用原生 `<blockquote>` 标签。

#### 2. 正在输入指示器时机（修复）
- **问题**：正在输入指示器仅在智能体处理完成后才出现，原因是 `agentExecutor.executeMessage()` 在输入协程启动前就阻塞了。
- **解决方案**：重新排列 `processInboundMessage()` 的执行顺序，使正在输入指示器协程在智能体执行前启动。智能体执行被包裹在 `scope.launch {}` 中以实现并发运行。

#### 3. 活跃会话集成（增强）
- **问题**：桥接消息通过 `preferences.getBridgeConversationId()` 发送到专用的桥接会话，与应用界面脱节。
- **解决方案**：桥接现在将消息路由到应用中当前活跃的会话。这意味着：
  - 消息发送到用户当前在 ChatScreen 中查看的会话。
  - `/clear` 创建新会话；ChatScreen 随即切换到该会话，并将其设为新的活跃目标。
  - 在应用中切换会话后，桥接消息立即跟随到新会话。
- `ConversationMapper` 不再依赖 `BridgePreferences` 进行会话 ID 存储。
- *注*：RFC-041 的初始实现通过数据库 `updated_at DESC` 追踪活跃会话，无法感知用户在 app 中的手动切换操作。此问题已在 RFC-045 中修复，通过 `BridgeStateTracker.activeAppSessionId`（由 `ChatViewModel` 在每次会话切换时更新）实现正确追踪。

#### 4. 纯文本回退（健壮性）
- **问题**：如果 HTML 渲染因任何原因失败，整个消息发送将无声地失败。
- **解决方案**：`TelegramChannel.sendResponse()` 现在将 HTML 渲染包裹在 try/catch 中。失败时回退到发送不带 `parse_mode` 的纯文本。

#### 5. 开机自动启动（此前已实现）
- `BridgeBootReceiver` 监听 `BOOT_COMPLETED` 事件，若设备关机前桥接服务已启用，则自动重启该服务。

#### 6. 前台通知持久化（此前已实现）
- 将 `foregroundServiceType` 从 `specialUse` 修正为 `remoteMessaging`，以获得正确的系统行为。（中间曾短暂使用 `dataSync`，但该类型在 Android 14+ 上有 6 小时时间限制；`remoteMessaging` 无此限制，且是持久消息服务的正确类型。）
- `ACTION_RESTART` intent 确保在频道重新加载时前台通知持续显示。

#### 7. 重复消息防止（此前已实现）
- 500 条消息的 LRU 去重缓存防止同一消息被处理两次，对长轮询频道至关重要。

#### 8. 桥接设置界面（此前已实现）
- 可折叠的频道区块，支持启用/禁用切换。
- 每个频道的分步设置指南。
- 唤醒锁切换开关，附带电池影响警告。
- 每个频道的访问控制白名单配置。

### 验收标准

1. Telegram 回复格式紧凑，段落之间只有一个空行（无多余空白）。
2. 用户发送消息时，Telegram 中立即显示正在输入指示器。
3. 桥接消息出现在应用最近使用的会话中，而非单独的桥接会话。
4. `/clear` 创建新会话，该会话成为后续桥接消息的活跃目标。
5. HTML 渲染失败时回退到纯文本，不影响消息正常发送。
6. 设备重启后，若桥接服务此前已启用，则自动启动。
7. 所有现有单元测试通过；新增测试覆盖格式变更。

### 范围之外
- 智能体向消息平台回传富媒体内容（图片、文件）。
- 多会话的逐频道映射（每个外部聊天对应不同会话）。
- 桥接消息的端到端加密。
